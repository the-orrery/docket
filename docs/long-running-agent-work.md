# 长线 agent 工作

长线 agent 工作不能只靠一条 Todo 承载。结构的目标是让人随时能回答三个问题：

1. 最终要达成什么？
2. 当前在哪个阶段？
3. 下一步最小安全动作是什么？

## 推荐拓扑

用 issue 树，不用扁平 backlog。

| 层级 | 示例 id | 用途 |
|---|---|---|
| 主线 | `ISSUE-100` | 总目标、非目标、最终 gate |
| 阶段 | `ISSUE-101` | 有边界的阶段和阶段出口 |
| 工作包 | `ISSUE-102` | 一次具体实现、审查或验证 |
| 证据评论 | `comments/ISSUE-102.md` | 命令结果摘要、决定、结论 |

主线 issue 应该保持短。易变的执行细节下沉到子 issue 和评论里，让顶层仍然
可读、可接手。

## 关键字段

| 字段 | 用法 |
|---|---|
| `parent` | 形成执行树 |
| `blocked_by` | 表达真实依赖，不表达模糊担心 |
| `batch` | 组织近期 Todo |
| `wake` | 暂停到某个日期 |
| `triage` | agent 自发提议先进入待审 |
| `project` | 归属到命名工作流 |

不要只把结构关系写在评论里。如果一个关系会影响 `active`、`ready` 或
`overview`，它应该变成字段。

## 阶段出口

每个阶段 issue 都应该说明：

| Section | 问题 |
|---|---|
| Goal | 阶段完成后发生什么变化 |
| Scope | 包含什么、不包含什么 |
| Gate | 进入下一阶段需要什么证据 |
| Risks | 什么情况会让阶段结论失效 |
| Next action | 当前最小下一步是什么 |

gate 必须可检查。“看起来不错”不够；优先使用具体本地证据，例如测试名、
扫描摘要、审查决定或文件清单。

## 状态报告

长线进度应该从树上报告：

```text
主线: ISSUE-100 发布准备
  Done: ISSUE-101 当前文件面清理
  In progress: ISSUE-104 文档脱敏改写
  Blocked: ISSUE-108 历史处理决策
  Next: 文档提交后重跑 current/history gate
```

这能给出紧凑状态，而不要求读者翻完所有评论。

## 收尾

工作包只有在实现和证据都存在后才关闭。阶段 issue 只有在子工作完成、取消，
或被明确移动到后续阶段后才关闭。

发布安全相关工作要把证据分开记录：

| 证据 | 含义 |
|---|---|
| 当前文件面扫描 | 当前文件里有什么 |
| 历史扫描 | 旧提交里还存在什么 |
| 人工确认 | 扫描器无法判断的项 |
| 后续 gate | 需要单独授权的动作，例如历史改写 |
